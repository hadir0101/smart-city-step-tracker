Section 01

Project Overview

What StepTrack does, why it was built, and how all the pieces fit together at a high level.

StepTrack is a full-stack web application that combines historical Apple Health data with custom real-time step detection from raw accelerometer data. It was built across four sequential phases, starting from simple data parsing and ending with a live sensor pipeline and algorithm validation.

The core motivation is to go beneath the iPhone's built-in pedometer black box. Instead of trusting an opaque step count, StepTrack collects raw 3-axis accelerometer readings from the Phyphox app and runs its own signal-processing algorithm to detect each footstrike — producing a step count that can be audited, tuned, and validated.

Database

How step data is stored, structured, and protected against duplicates.

The project uses SQLite — a file-based database that requires no server installation. Everything lives in a single file called steps.db. This is appropriate for a prototype: the file can be copied, emailed, opened in any SQLite browser, or version-controlled.

Schema — step_records table

idINTEGERPRIMARY KEY AUTOINCREMENTUnique row identifier, assigned automatically
start_dateTEXTNOT NULLOriginal timestamp string from the Apple Health XML (e.g. "2026-04-15 08:12:03 +0200")
dateTEXTNOT NULLYYYY-MM-DD in Europe/Paris timezone — used for grouping and daily queries
hourINTEGERNOT NULLHour of day 0–23 in Europe/Paris timezone — enables hourly breakdown
stepsREALNOT NULLStep count for this interval (REAL allows fractional values from Apple Health)
sourceTEXT—"iPhone", "phyphox", or "manual" — tracks where each record came from

UNIQUE(start_date, source) — If you run the ingestion script twice, duplicate records are silently skipped (INSERT OR IGNORE). The same raw data can never appear twice in the database.

Key Functions in database.py

get_db()

Opens a SQLite connection with row_factory = sqlite3.Row so results can be accessed like dictionaries (e.g. row['steps']).

init_db()

Creates the step_records table using CREATE TABLE IF NOT EXISTS — safe to call on every startup. Does nothing if the table already exists.

seed_from_xml()

Parses export.xml from Apple Health, converts all timestamps to Paris time, and inserts records. Skips entirely if the DB already has data.

Section 04

Step Detection Algorithm

A complete explanation of how raw accelerometer numbers become a step count — the science, the code, and the design choices.

Why Raw Accelerometer Data?

The iPhone's built-in pedometer uses a dedicated motion coprocessor (Apple M-series chip) and a proprietary closed algorithm — you receive only a final step count. There is no way to inspect or validate what it does internally.

By capturing the raw 3-axis linear acceleration signal from Phyphox, we can implement detection from scratch, understand every parameter, audit every decision, and tune the algorithm to match observed walking behavior.

Understanding the Raw Signal

The accelerometer measures linear acceleration along three axes simultaneously:

  • ax — left/right acceleration (m/s²)
  • ay — forward/backward acceleration (m/s²)
  • az — up/down acceleration (m/s²)

Each walking step produces a brief burst of acceleration as the foot strikes the ground and the body's weight transfers. This shows up as a sharp spike in the signal. The problem is that the spike direction depends entirely on how the phone is oriented in your pocket or hand — a step might appear mostly in ax on one person and mostly in az on another.

Step 1 — 3-Axis Vector Magnitude

📡

Why: Remove orientation dependency

Instead of choosing one axis, we compute the overall strength of acceleration across all three axes at once. This is the Euclidean vector magnitude — the length of the 3D acceleration vector.

# For every sample, compute the total acceleration magnitude
df['magnitude'] = np.sqrt(
    df['ax']**2 + df['ay']**2 + df['az']**2
)

The result is a single number per sample that answers "how much total acceleration is happening right now?" — regardless of which direction. The phone can be upside down, sideways, or flat, and each footstrike still produces the same magnitude spike.

Mathematical intuition: If ax=0, ay=0, az=5 (step upward), magnitude = 5. If ax=3, ay=4, az=0 (step sideways), magnitude = 5. Same step, same result — orientation removed.

Step 2 — Butterworth Low-Pass Filter

〰️

Why: Remove noise without distorting the walking signal

Raw accelerometer data contains high-frequency noise — electrical sensor noise, micro-vibrations from the phone case, even interference from the phone's own components. Walking itself is a slow, rhythmic process. We need to separate these two things.

Walking frequency: A typical adult takes 80–140 steps per minute, which is 1.3–2.3 steps per second, or 1.3–2.3 Hz. The raw signal noise lives mostly above 5–10 Hz. We choose a cutoff of 3 Hz — this keeps the full walking signal and discards everything faster.

Why Butterworth? The Butterworth filter has a maximally flat frequency response in the passband — meaning it does not amplify or distort the walking signal at all before the cutoff, then smoothly rolls off above it. Other filter types (Chebyshev, elliptic) are sharper but introduce ripple that can create false peaks.

Why order 4? A higher-order filter gives a steeper rolloff — it more aggressively removes high-frequency noise. Order 4 is a practical sweet spot: steep enough to clean the signal, but stable and well-behaved numerically.

Why filtfilt? A standard filter causes phase distortion — peaks in the filtered signal are shifted in time relative to the original. This would move detected footstrikes away from the true moment of impact. filtfilt applies the filter twice — once forward, once backward — which cancels the phase shift exactly. Peaks stay aligned with the original signal.

# Compute the Nyquist frequency (half the sampling rate)
nyq = 0.5 * fs                           # e.g. 0.5 × 50 = 25 Hz

# Design a 4th-order Butterworth low-pass filter at 3 Hz
b, a = butter(4, 3.0 / nyq, btype='low')   # b, a are filter coefficients

# Apply zero-phase filtering (forward + backward pass)
df['filtered'] = filtfilt(b, a, df['magnitude'])

Step 3 — Adaptive Peak Detection

▲

Why: Each local maximum in the filtered signal = one footstrike

After filtering, the signal shows one smooth hump per footstrike. scipy's find_peaks locates all local maxima above a threshold that are sufficiently far apart in time.

Two key parameters control detection quality:

Height threshold — adaptive, not fixed:

threshold = df['filtered'].mean() + df['filtered'].std() * 0.4

The threshold is set at mean + 0.4 × standard deviation of the filtered signal. This is adaptive — it automatically scales with how hard the person is walking. Someone walking gently has a lower-amplitude signal; someone walking fast has a higher one. A fixed number (like 1.0 m/s²) would fail on gentle walks or over-detect on vigorous ones. By tying the threshold to the signal's own statistics, it adjusts automatically to each session.

The 0.4 multiplier was tuned empirically: high enough to avoid counting background noise as steps, low enough to catch every real footstrike even during gentle walking.

Minimum distance between peaks:

min_gap = int(fs * 0.3)   # e.g. 50 Hz × 0.3s = 15 samples minimum gap

peaks, _ = find_peaks(
    df['filtered'],
    height=threshold,      # ignore anything below this
    distance=min_gap       # peaks must be at least 0.3s apart
)

The 0.3-second minimum gap enforces a maximum cadence of 200 steps/min — which is faster than any realistic walking pace. Without this, a single footstrike that causes two close peaks in the signal (a common artifact) would be counted as two steps. By requiring 0.3s between peaks, we prevent all such double-counts.

Step 4 — Results and Cadence

✅

Step count, duration, and cadence

The number of detected peaks is the step count. Cadence is derived from that count and the session duration.

step_count = len(peaks)

duration   = float(df['time'].iloc[-1] - df['time'].iloc[0])   # seconds

cadence    = step_count / (duration / 60.0)                 # steps per minute

The result dictionary also contains the sampling rate fs, which is estimated automatically from the median time difference between samples (rather than assuming 50 Hz), making the algorithm robust to irregular sampling.

Complete Detection Function

This is the entire core detection logic in one place — _run_detection(df) in step_detection.py:

def _run_detection(df):
    df = df.copy()

    # Step 1: 3-axis vector magnitude
    df['magnitude'] = np.sqrt(df['ax']**2 + df['ay']**2 + df['az']**2)

    # Auto-detect sampling rate from median inter-sample interval
    dt = df['time'].diff().median()
    fs = round(1.0 / dt) if dt > 0 else 50.0

    # Step 2: Butterworth low-pass filter at 3 Hz (zero-phase)
    df['filtered'] = _butter_lowpass(df['magnitude'], cutoff=3.0, fs=fs)

    # Step 3: Adaptive threshold + peak detection
    threshold = df['filtered'].mean() + df['filtered'].std() * 0.4
    min_gap   = int(fs * 0.3)
    peaks, _  = find_peaks(df['filtered'], height=threshold, distance=min_gap)

    # Step 4: Results
    step_count = len(peaks)
    duration   = float(df['time'].iloc[-1] - df['time'].iloc[0])
    cadence    = step_count / (duration / 60.0) if duration > 0 else 0.0

    return {
        'df':         df,         # DataFrame with magnitude + filtered columns added
        'peaks':      peaks,      # array of sample indices where steps were detected
        'steps':      step_count,
        'duration_s': round(duration, 1),
        'cadence':    round(cadence, 1),
        'fs':         fs,
        'threshold':  round(threshold, 4),
    }

Section 05

Cadence-Based Validation

How the algorithm validates its own output using physiological walking norms.

Why Cadence, Not Direct Comparison?

The natural instinct is to compare our detected step count against Apple Health. But this has a fatal problem: different time windows. A 2-minute Phyphox session detects ~216 steps. Apple Health shows 7,700 steps for the full day of April 15th. Comparing these two numbers would give a "97% error" — which is completely meaningless because we are not measuring the same thing.

The correct validation is to check whether the detected cadence (steps per minute) falls within the physiologically established range for normal human walking. This is independent of session length.

GOOD

80–140 steps/min

Normal walking pace. Detection is working accurately. Most adults walk between 100–120 steps/min.

ACCEPTABLE

60–80 or 140–180 steps/min

Slow stroll or brisk/speed walking. Slightly outside typical range but physiologically plausible.

NEEDS TUNING

< 60 or > 180 steps/min

Outside the human range. Algorithm is either missing steps or over-counting. Adjust threshold or filter cutoff.

cadence = detection['cadence']

if 80 <= cadence <= 140:
    verdict = 'GOOD'
elif 60 <= cadence <= 180:
    verdict = 'ACCEPTABLE'
else:
    verdict = 'NEEDS TUNING'

Apple Health as a Reference (Not a Target)

When a date is provided to the detection pipeline, the Apple Health total for that day is fetched from the database and shown alongside the detection result. It serves as context — you can see whether the person was generally active that day — but it is not used to calculate the verdict. The label in the dashboard says "Apple Health (day total)" to make this distinction explicit.

Section 06

The Two Ingestion Pipelines

Both pipelines share the same detection algorithm — only the data source differs.

Section 1 — CSV Ingestion (ingestion/csv_ingest.py)

This pipeline processes a Phyphox CSV export file that already exists on disk. No network connection to the phone is needed.

Data flow: CSV file → load_phyphox() → detect_steps() → cadence verdict → optional Apple Health reference → return result dict

The load_phyphox() function in step_detection.py handles a subtlety: different Phyphox experiments produce different column names. The "Acceleration (without g)" experiment labels columns as "Acceleration x (m/s^2)", while "Linear Acceleration" labels them "Linear Acceleration x (m/s^2)". The function checks for both 'x' in c AND 'acceleration' in c (case-insensitive) to handle both variants, plus short aliases ax/ay/az for data generated programmatically.

# ingestion/csv_ingest.py — main entry point
def run(csv_path='phyphox_data.csv', date_str=None, ...):

    if not os.path.exists(csv_path):
        raise FileNotFoundError(...)      # clear error if file is missing

    detection = sd.detect_steps(csv_path)   # run full algorithm

    cadence = detection['cadence']
    verdict = 'GOOD' if 80 <= cadence <= 140 else \
              'ACCEPTABLE' if 60 <= cadence <= 180 else 'NEEDS TUNING'

    if date_str:
        apple = sd.apple_health_count(date_str)   # query DB for reference

    return { 'steps': ..., 'cadence': ..., 'validation': { 'verdict': verdict }, ... }

Command-line usage:

python -m ingestion.csv_ingest phyphox_data.csv 2026-04-15

Section 2 — Live API Ingestion (ingestion/live_ingest.py)

This pipeline connects to a running Phyphox instance on the phone and streams accelerometer data in real time while you walk.

Data flow: Connect to phone → auto-discover buffer names → poll every 500ms → accumulate DataFrame → detect steps → verdict → save timestamped CSV → return result

discover_channels()

Before any data collection, the function calls GET http://phone:8080/config. Phyphox returns a JSON object listing all buffer names for the current experiment. The function searches those names by keyword ('accx', 'linx', 'x' etc.) to automatically find the correct channel names — so the pipeline works with any Phyphox experiment without manual configuration.

collect()

Polls GET http://phone:8080/get?linX=<last_t>&linY=<last_t>&linZ=<last_t> every 500ms. The last_t parameter tells Phyphox to return only samples newer than that timestamp — no duplicates. Each response is a JSON object with buffers of new x/y/z values. After the collection window ends, all accumulated arrays are assembled into a pandas DataFrame and passed to detect_steps_from_df().

# Simplified polling loop
all_t, all_ax, all_ay, all_az = [], [], [], []
last_t   = 0
deadline = time.time() + duration_s

while time.time() < deadline:
    t_buf, ax_buf, ... = _poll(get_url, ch_x, ..., last_t)  # new samples only
    all_t.extend(t_buf)
    last_t = t_buf[-1]
    time.sleep(0.5)                                           # poll every 500ms

df = pd.DataFrame({'time': all_t, 'ax': all_ax, ...})

Mock Server (ingestion/mock_server.py)

Phyphox remote access can fail due to iOS Local Network permissions, phone firewall settings, or Private WiFi Address obfuscation. The mock server is a Flask application on port 8080 that serves phyphox_data.csv through the identical HTTP API that Phyphox uses.

GET /config

Returns a JSON object with linX, linY, linZ, time as buffer names — exactly what Phyphox returns.

GET /get

Reads the CSV and streams chunks of data in real time at the correct sampling rate, responding only with samples newer than last_t.

GET /

Status page showing elapsed time, samples served, and progress percentage — useful for monitoring a running demo.

# Start the mock server (Terminal 1)
python -m ingestion.mock_server

# Then run the live pipeline against it (Terminal 2)
python -m ingestion.live_ingest 127.0.0.1 30 2026-04-15

Key Design Decisions — Q&A Preparation

Why SQLite?

No server to install. Single file — portable, copyable, version-controllable. Sufficient for a single-user prototype with thousands of records.

Why cadence validation?

A 2-minute session cannot be compared against a full-day total. Cadence validation is fair regardless of session length — it checks if the walking pace is humanly plausible.

Why vector magnitude?

Phone orientation is unpredictable. √(ax²+ay²+az²) gives the same result regardless of which way the phone faces — orientation-invariant by construction.

Why filtfilt?

A standard one-pass filter shifts peaks in time. filtfilt applies the filter twice (forward + backward), cancelling all phase shift. Peaks stay at the correct timestamps.

Why adaptive threshold?

A fixed threshold (e.g. 1.0 m/s²) fails on gentle walks or high-amplitude runs. Mean + 0.4×std adapts automatically to each session's signal strength.

Why a mock server?

iOS blocks local network access without explicit permission, which can silently fail. The mock server makes the live pipeline fully demonstrable without phone dependency.
